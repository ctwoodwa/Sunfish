## A1 (Critical) — Phase 3 canonicalization: pin existing `Sunfish.Foundation.Crypto.CanonicalJson` (Option 2)

The original hand-off Phase 3 referenced `Sunfish.Foundation.Canonicalization.JsonCanonical` per ADR 0054 A1. Council verified (`git ls-tree -r origin/main | grep Canonicalization` returns zero matches in `Sunfish.Foundation.Canonicalization.*`): **the cited symbol is vapourware**. ADR 0054 A1 promised the new namespace + new `packages/foundation-canonicalization/` package; its implementation checklist is unchecked on `origin/main`. Existing canonicalizers are explicitly NOT RFC 8785 conformant — `JsonCanonicalCanonicalizer.cs` self-documents this in its XML remarks.

**Resolution per council A1 Option 2 (minimal-resequencing):** Phase 3 builds against the existing pragmatic canonicalizer. Strict RFC 8785 conformance is deferred to a follow-up amendment when ADR 0054 A1's foundation-canonicalization package actually ships.

Replace Phase 3's canonicalizer references with:

> **Canonicalizer parity target:** `Sunfish.Foundation.Crypto.CanonicalJson.Serialize` (existing on `origin/main` at `packages/foundation/Crypto/CanonicalJson.cs`). The existing pragmatic-canonicalizer rules are: alphabetical key sort, no whitespace, UTF-8 encoding, `System.Text.Json` number serialization. Cross-language test fixtures generated by `CanonicalJson.Serialize` (NOT `JsonCanonicalCanonicalizer` or `JsonCanonicalizer` — those are also pragmatic but produce different bytes; pin to one).

> **Forward path:** when ADR 0054 A1's promised `packages/foundation-canonicalization/Sunfish.Foundation.Canonicalization.JsonCanonical` ships (likely as a W#21 follow-up phase OR a standalone hand-off), Phase 3 (and W#21 + ADR 0054's signature canonicalization) re-target the strict RFC 8785 canonicalizer in a forward-compat amendment. Phase 2.1 ships against `CanonicalJson` parity; Phase 2.2+ migrates to RFC 8785 strict.

**Phase 3 acceptance criteria revised** (per A4 below).

## A4 (Major) — Phase 3 acceptance criteria: tightened

Original hand-off Phase 3 lists "10/10 cross-language canonicalization fixtures match byte-for-byte" without specifying fixture types or generation pipeline. Tightened per council A4:

**Replace Phase 3 acceptance criteria with:**

- [ ] Fixture types covered (10+):
  - Number edge cases: very large integers (≥2^53), leading zeros (`0001` rejected per JSON spec), scientific notation (`1.5e10`), negative zero (`-0`), `NaN` / `Infinity` rejection
  - Nested objects with key-order variation (input keys in random order; output canonical-sorted)
  - Arrays with mixed types (`[1, "two", null, true, {}]`)
  - UTF-8 strings with non-ASCII (`"café"`, `"日本語"`, `"🎉"`)
  - JSON literal values (`null`, `true`, `false`)
  - Empty containers (`{}`, `[]`)
  - Deeply-nested structures (≥10 levels)
- [ ] .NET-side canonical bytes generated by `Sunfish.Foundation.Crypto.CanonicalJson.Serialize` (pin per A1; do NOT use `JsonCanonicalizer` or `JsonCanonicalCanonicalizer` — they produce different bytes)
- [ ] Fixture format: `(input.json, expected_canonical.bin)` byte-pairs (binary not text — avoid encoding ambiguity)
- [ ] Fixture file location: `_shared/fixtures/json-canonical/` — note: this directory does NOT exist on `origin/main`; Phase 3 creates it
- [ ] Cross-language test pinned to byte-identity comparison (not string comparison; not "equivalent" comparison)

## A5 (Major) — Phase 2 SQLCipher key-loss = queued-event loss: accepted risk

Original Phase 2 ships single-device-bound SQLCipher key derivation with no recovery path. Council noted the gap: device loss / Keychain wipe / failed iCloud-Keychain-migration → unrecoverable local SQLCipher database → queued-but-unsynced events lost. Foundation.Recovery's `PaperKeyDerivation` pattern doesn't trivially port to mobile.

**Accepted risk (Phase 2.1):** name the trade-off + add user-visible first-launch warning. **NOT** a halt-condition (the alternative — mobile recovery surface — would be a substantial ADR + new phase set; out of substrate-v1 scope).

Add the following sub-section to Phase 2:

> **Phase 2 accepted risk: device loss = queued-event loss.** SQLCipher is keyed by a Phase-1-only-key derived from the install root + sentinel `"sunfish-field-sqlcipher-v1"`. The derivation has no recovery path: if the iPad's Keychain entries are wiped (factory reset, hardware loss, OS migration that fails to migrate Keychain), the local database is unrecoverable — including any events captured but not yet synced to Anchor.
>
> **Mitigation (Phase 2.1):** user-visible first-launch warning shown once when pairing completes:
>
> > "Sunfish Field stores captured events on this device until they sync to your Anchor. **If you lose this device or reset it before sync completes, those events are unrecoverable.** Sync frequently — your queue status is on the home screen. Tap to force-sync at any time."
>
> Show the warning via a SwiftUI sheet on the post-pairing flow; require explicit user acknowledgment (tap "I understand"). Audit-emit `FieldDeviceFirstLaunchWarningAcknowledged` (new AuditEventType — add to A2.10 the 5 AuditEventType list per Phase 5) so the acknowledgment is auditable.
>
> **Forward path:** mobile recovery surface (deferred mobile-paper-key, iCloud-Drive blob backup as opt-in, cross-device pairing-promotion to allow restoring queue from a paired sibling device, etc.) is a Phase 2.2+ ADR. Author when there is a real customer scenario hitting this — currently speculative.

**Update Phase 5 audit-event count** from 4 to 5 (add `FieldDeviceFirstLaunchWarningAcknowledged`).

## A6 (Minor) — Phase 1 Info.plist corrections

Original hand-off Phase 1 specified `UIRequiredDeviceCapabilities = [armv7]` — invalid for iOS 16+ baselines (armv7 is iOS 7-era 32-bit ARM; removed when 32-bit support ended in iOS 11). Plus Phase 1 missed several Apple-required Info.plist entries.

Replace Phase 1's Info.plist spec with:

```xml
<key>CFBundleIdentifier</key><string>dev.sunfish.field</string>
<key>CFBundleVersion</key><string>1</string>
<key>CFBundleShortVersionString</key><string>0.1.0</string>
<key>MinimumOSVersion</key><string>16.0</string>
<key>LSRequiresIPhoneOS</key><true/>
<key>UILaunchScreen</key><dict>
    <key>UIColorName</key><string>LaunchScreenBackground</string>
</dict>
<key>NSCameraUsageDescription</key><string>Sunfish Field uses the camera to capture receipts, equipment nameplates, and inspection photos.</string>
<key>NSPhotoLibraryUsageDescription</key><string>Sunfish Field accesses your photo library to attach existing photos to captured records.</string>
<key>BGTaskSchedulerPermittedIdentifiers</key><array>
    <string>dev.sunfish.field.upload</string>
</array>
```

NOTE: `UIRequiredDeviceCapabilities` is **omitted entirely** — the iOS App Store derives device support from the binary's architectures + `MinimumOSVersion`. If a future scenario requires explicit capabilities (e.g., `arkit`, `gps`), add them then.

**Privacy manifest required** (Apple-mandated since iOS 17 for new app submissions):

- Add `accelerators/anchor-mobile-ios/SunfishField/PrivacyInfo.xcprivacy` declaring:
  - `NSPrivacyAccessedAPITypes` — declare any `NSPrivacyAccessedAPICategoryUserDefaults`, `NSPrivacyAccessedAPICategoryFileTimestamp`, `NSPrivacyAccessedAPICategoryDiskSpace`, etc. that the GRDB / URLSession / Keychain code uses (consult Apple's Required Reasons API list for current required-reason codes)
  - `NSPrivacyTrackingDomains` — empty array (Sunfish Field does not track)
  - `NSPrivacyCollectedDataTypes` — declare what's captured (location? photos? user-account-id?) per Apple's data-collection list

Phase 1 acceptance gate adds: `[ ] PrivacyInfo.xcprivacy validates against Xcode's privacy-manifest schema`.

## A7 (Minor) — `DeviceId` truncation: name Phase 3+ growth path

Original Phase 0 specifies `DeviceId` as "first 16 hex chars of SHA-256 of the public key." 16 hex chars = 64-bit truncated SHA-256 → birthday-bound collision probability ~1 in 2^32 across the install population. Fine for Phase 2.1 single-tenant property business (≤10 paired devices); not fine for Phase 2.2+ multi-tenant deployments with thousands of paired iPads.

Add note to Phase 0's `DeviceId.swift` spec:

> **Phase 2.1 truncation:** 16-hex-char SHA-256 prefix is sufficient for single-tenant property-business scope (≤10 paired devices; collision probability negligible). **Phase 3+ growth path:** Phase 2.2+ multi-tenant deployments may require widening to full SHA-256 hex (64 chars) OR full Ed25519 public-key hex (64 chars) at the cost of a longer envelope. Migration is forward-compatible — existing 16-char `device_id` values remain valid; new captures may use the wider format with an explicit `device_id_format_version` envelope field added in the rotation amendment.

## A8 (deferred to first build, not addendum-required)

Phase 4 `BGTaskScheduler` lifecycle: Phase 4 implies `URLSessionConfiguration.background` for uploads but does NOT name the lane separation between `BGTaskScheduler` (budget ~30 sec/run, ~1 run/day pessimistic; runs on charge + Wi-Fi) and `URLSessionConfiguration.background` (orthogonal lane; OS-managed retry; consumes neither BGTaskScheduler budget). Combined the design works, but Phase 4 spec doesn't make the separation explicit. **Recommendation:** Phase 4 spec should clarify that:

- `URLSessionConfiguration.background` is the **upload lane** (per ADR 0028-A2.2 settings — `discretionary=false` = uploads run when triggered, not deferred)
- `BGTaskScheduler` is the **opportunistic-maintenance lane** (`BGProcessingTaskRequest` for blob compaction, queue VACUUM, etc.; `discretionary=true`-equivalent)

Defer this clarification to Phase 4 first-build PR rather than authoring it here. If COB hits ambiguity, write `cob-question-*-w23-p4-bgtask-lane.md` and XO authors a follow-up addendum (or A9 here).
